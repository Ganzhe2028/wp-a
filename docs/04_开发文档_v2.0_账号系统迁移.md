## 1. 这一版改什么（v1.0 → v2.0）

| 维度 | v1.0（现状） | v2.0（目标） |
|---|---|---|
| 浏览者身份 | 无身份 | 登录后有身份（账号） |

## 5. 身份与权限（v2.0）

三类身份并存：

1. **学生**：用户名+密码登录 → 服务端签发 `owk_session`（httpOnly JWT，载 `personId`）。所有 `/me`、`/u`、`/loc`、收藏接口都认这个 cookie。无 session 一律跳登录（带 `next` 回跳，见 §10）。
2. **管理员**：维持现状，`ADMIN_PASSWORD` → `owk_admin` cookie，admin 接口校验它。**与学生 session 完全独立**，两套 cookie 互不影响。
3. `editToken`：退役。列保留但主流程不用；`/edit/[token]` 路由可删。
   > ⚠️ **别半删。** `editToken` 列仍是 `String @unique` **非空**约束，所以 **import 必须继续调用 `createUniqueEditToken()` 生成它**，哪怕没人用——否则建账号会撞非空约束、create 直接失败。要区分两件事：① **DB 列**（仍非空，仍要生成）；② **输出层**——`/edit/[token]` 路由、export CSV 里的 `edit` 链接列，这些可以删，跟 DB 列无关。若想彻底清掉这列，**单独**做一次把它改成 `String?` 的迁移，别和本次混在一起。

学生 session（加进 `lib/auth.ts`，复用已有 `jose`，照搬 admin cookie 那套，只是更长效）：

```ts
function getRequiredEnv(name: string): string {
  const value = process.env[name]?.trim();
  if (!value) throw new Error(`Missing required environment variable: ${name}`);
  return value;
}

function getSessionSecret(): Uint8Array {
  return new TextEncoder().encode(getRequiredEnv("SESSION_SECRET"));
}

const SESSION_COOKIE = "owk_session";
const SESSION_MAX_AGE = 60 * 60 * 24 * 14; // 14 天 > 活动 10+ 天

export async function createStudentSession(personId: string): Promise<string> {
  return new SignJWT({ pid: personId, role: "student" })
    .setProtectedHeader({ alg: "HS256" })
    .setIssuedAt()
    .setExpirationTime("14d")
    .sign(getSessionSecret());
}

export async function verifyStudentSession(): Promise<{ personId: string } | null> {
  const token = (await cookies()).get(SESSION_COOKIE)?.value;
  if (!token) return null;
  try {
    const { payload } = await jwtVerify(token, getSessionSecret());
    return { personId: payload.pid as string };
  } catch {
    return null;
  }
}

export function setStudentCookie(token: string) {
  return {
    name: SESSION_COOKIE, value: token,
    httpOnly: true,                                  // ← 关键，见 §7
    secure: process.env.NODE_ENV === "production",
    sameSite: "lax" as const, path: "/",
    maxAge: SESSION_MAX_AGE,
  };
}

export function clearStudentCookie() {
  return { name: SESSION_COOKIE, value: "", httpOnly: true, path: "/", maxAge: 0 };
}
```

新增环境变量：`SESSION_SECRET=`——**至少 32 字节（256 bit）的随机串**（如 `openssl rand -base64 48` 生成）。jose 用它做 HMAC 签名，且 **jose 不会自动校验 key 长度**，太短会埋安全隐患。Vercel 也要填。（现有 admin 用 `ADMIN_PASSWORD` 当 HMAC 密钥，同理建议够长。）`ADMIN_PASSWORD` 和 `SESSION_SECRET` 都必须配置；auth 会懒读取这些变量，缺失时相关 session 操作 fail closed，不允许硬编码 fallback。

## 6. 密码哈希（node:crypto，不加库）

栈是冻结的，**不要装 bcrypt/argon2**。用 Node 内置 `scrypt`。放 `lib/auth.ts`：

```ts
import { randomBytes, scryptSync, timingSafeEqual } from "node:crypto";

export function hashPassword(plain: string): string {
  const salt = randomBytes(16);
  const hash = scryptSync(plain, salt, 64);
  return `${salt.toString("hex")}:${hash.toString("hex")}`;
}

export function verifyPassword(plain: string, stored: string): boolean {
  const [saltHex, hashHex] = stored.split(":");
  if (!saltHex || !hashHex) return false;
  const salt = Buffer.from(saltHex, "hex");
  const expected = Buffer.from(hashHex, "hex");
  const actual = scryptSync(plain, salt, expected.length);
  return expected.length === actual.length && timingSafeEqual(expected, actual);
}
```

`scryptSync` 是同步的，登录是低频操作，serverless 下可接受。比较必须用 `timingSafeEqual`，不要用 `===`。

> **运行时**：用到 `node:crypto` 的路由（登录、重置密码、生成账号）必须跑在 **Node runtime**，不要给它们设 `export const runtime = 'edge'`——Edge 上没有 `node:crypto`。App Router 路由处理器默认就是 Node runtime，保持默认即可。（Codex 提到 bcryptjs 也是一个选项，但那是新依赖，`AGENTS.md` 冻结了栈，scrypt 内置且同样安全，故沿用 scrypt；要破例加 bcryptjs 得先跟管 AGENTS.md 的人确认。）

可打印密码生成（`lib/code.ts`，使用独立字母表）：

```ts
const PRINTABLE_PASSWORD_ALPHABET =
  "ABCDEFGHJKLMNPQRSTUVWXYZabcdefghjkmnpqrstuvwxyz23456789!@#$%^&*-_+=?";
const generatePassword = customAlphabet(PRINTABLE_PASSWORD_ALPHABET, 12);
export function newPlainPassword(): string { return generatePassword(); }
```

12 位、大小写 + 数字 + 少量安全符号，排除易混淆和难手输字符。提供约 69^12 种组合，配合 scrypt hash 可抗离线暴力破解；纸条排版需预留足够宽度。

## 7. 会话保持：撑过 10+ 天（关键，必读）

「登一次管 10+ 天」整个压在 cookie 上。两条硬要求：

1. **必须是服务端 `Set-Cookie` 下发的 httpOnly cookie。**
   - iOS Safari 的 ITP 会把**脚本写的 cookie**（`document.cookie`）和 localStorage 砍到约 7 天。服务端下发的 httpOnly cookie **不**受这条 7 天限制约束——这正是选它的原因。
   - 所以：session **只能**走 §5 的 `setStudentCookie`（服务端 `Set-Cookie`）。**禁止**把登录态存进 localStorage、`document.cookie` 或任何前端 JS 写的存储。
   - `owk_admin` 已经是这个模式，照抄即可。
2. **真机验证，别只测当天。** 在真 iPhone Safari 上：登一次 → 隔几天再开 `/u/xxx` → 确认仍是登录态。活动 10+ 天，至少要确认能跨过第 7 天。Safari 的 ITP 细则会随版本变，上线前按当前版本再核一遍，别只信本文。（cookie 的 Max-Age 是上限、不是保证——RFC 6265 允许浏览器因存储压力等提前清；实践中主流平台都撑得住，但正因为不是保证，真机验证是必须项不是可选项。）

> 失败长这样：不是「忘密码」，而是「明明登过却被登出、手边又没纸条」。被登出期间碰的收藏会丢账。所以第 1 条务必做对。

## 8. 收藏（服务端，单向，静默）

- 账号之间唯一的交互就是收藏。**单向**：A 收藏 B，只进 A 的列表；B 那边什么都不发生。
- 不通知 B、不给 B 看次数、不给 B 看是谁。系统里不存在「谁收藏了我」的任何接口或页面。
- 存数据库（`Favorite` 表），不存 localStorage——因为现在收藏者是已登录账号，身份已知，存库顺理成章，也不会被 Safari 清掉。
- 自己不能收藏自己（`favoriteeId === session.pid` 直接拒）。
- 收藏按钮初始状态：`/u`、`/loc` 是 server component，渲染时按 session 查这一对是否已存在，把布尔值传给客户端按钮，避免闪烁。

> **不做 localStorage 缓存（明确决定）。** 有 review 会建议「本地缓一份收藏、读得快、能离线、跨设备恢复」。这套设计不需要：页面在登录闸之后才进、本来就联网且要查库渲染主页，收藏顺带一个查询，150 人规模毫无压力；服务端是唯一真相源，换手机登录后收藏自然还在。而加本地缓存就要处理本地与服务端不一致的合并逻辑，是净增复杂度和新的丢账面。要「点一下立刻变心形」用乐观更新即可（先改按钮状态、后台发 POST、失败回滚），不要 localStorage。

## 9. 账号预生成与帐密下发（运营侧最关心）

流程：运营提前拿到 100+ 名单 → 开发跑一次生成 → 每人一条 `username + 明文密码` → 印在录取通知书/纸条上 → 学生当天照着输。

实现要点：

- `username` **默认 = 该人的 `code`**（已唯一、已去混淆、本来就要印）。学生纸条上就两样：用户名(=code) + 密码。如运营想要更友好的用户名，可在 import 时传入覆盖。
- import（或一个生成步骤）对每行：建 Person 时生成 `username`、`newPlainPassword()`，存 `passwordHash = hashPassword(明文)`，**把明文放进响应返回**。批量导入必须事务化；任一行失败时整批回滚，不返回半成功账号。
- ⚠️ **明文只在生成那一刻拿得到。** 存库的是 hash，事后无法还原明文。所以：
  - `/api/admin/import` 的响应必须带明文：`created: [{ chineseName, code, username, password }]`。
  - admin 页在导入成功后立刻提供「下载帐密 CSV」，运营当场存下、去印。错过这次就只能重置。
  - `/api/admin/export`（事后导出）**只能**给 `username`，给不了密码——别在这里假装能导密码。
- 忘密码走人工：`POST /api/admin/reset-password { code }` → 生成新明文、更新 hash、**响应返回新明文一次**。组长现场联系开发，开发重置后把新密码报给组长。不做学生自助找回。

## 10. 登录闸与回跳

- `/me`、`/u/[code]`、`/loc/[code]`：server component 开头 `verifyStudentSession()`，为空则 `redirect('/?next=' + encodeURIComponent(当前完整路径))`。
- Landing `/`：未登录显示学生登录表单（用户名+密码）+ 角落「管理员入口」链到 `/admin`；已登录则 `redirect('/me')`。
- 登录成功后：若有合法 `next`（站内路径，必须校验以防开放重定向）则跳 `next`，否则跳 `/me`。
- 效果：学生当天先在 `/` 登一次；之后任何一次碰 NFC 弹出的 `/u` 或 `/loc`，cookie 在就直接看；登录态不在（如还没登）就被弹去登录、登完回到刚碰的那个人。**回跳必须做对，否则「先碰了某人再登录」会丢掉那一碰。**

> 备选（不默认做）：若后续希望 `/loc`（纯找座位指路）登录前也能看，可只在点收藏时才要求登录。当前按你的明确要求：两页都登录闸。

## 12. 新增 Gotchas（在 AGENTS.md 既有清单之外）

- **session 只能服务端 httpOnly 下发**，不准落 localStorage / `document.cookie`（§7）。这是活动期不掉线的命根子。
- **登录和每次碰 NFC 必须同一个浏览器。** cookie 不跨浏览器。最常见翻车：在**微信内置浏览器**里登的录，碰 NFC 弹出的却是 Safari（两个独立 cookie jar），看起来就像没登过。处理：检测微信 UA（`navigator.userAgent` 含 `MicroMessenger`）时显示引导「请点右上角，在 Safari/默认浏览器中打开」，全程统一用系统默认浏览器。另外两条同方向的坑：① 微信内置浏览器里**别在活动期间登出微信账号**——登出会连带清掉 cookie 和 localStorage；② UC 浏览器内核太旧（约 Chrome 123），SameSite 行为可能有差异。这两条都靠「统一用系统默认浏览器」一条引导规避。
- **明文密码只在生成那一刻可得**，事后只有 hash（§9）。import 必须当场返回并让运营下载，别指望 export 能补。
- **收藏单向且静默**：不得新增任何「谁收藏了我」的读路径（§8）。`favoritesReceived` 反向关系只为级联删除。
- **回跳要校验 `next` 是站内路径**，防开放重定向。
- **主页可见性由系统自动决定**：学生上传头像并保存资料后，`/api/me` PATCH 自动把 `published` 设为 `true`；学生端无 toggle。运营仍可用 `hidden` 下架。
- **登录接口已加 rate limiting**：同一 IP 15 分钟内最多 20 次，同一用户名最多 5 次，防止在线暴力破解。
- **图片记录必须保存 `/api/upload-url` 返回的真实 `key`**：不要从 `publicUrl` 反推。`/api/me/images` 会校验 key 属于当前学生、URL 与 key 匹配，并在 Serializable transaction 内做最终数量上限检查。
- **frozen stack**：密码哈希用 `node:crypto`，不加库。
- **两套 cookie 独立**：`owk_session`（学生）与 `owk_admin`（运营）互不干扰，别合并。
- **Prisma migrate 遇到已有数据时**：加 NOT NULL 列到非空表要手动三步走——`ALTER TABLE ... ADD COLUMN`（nullable）→ `UPDATE ... SET`（回填）→ `ALTER TABLE ... ALTER COLUMN SET NOT NULL` + `CREATE UNIQUE INDEX`。`prisma migrate dev` 此时报错是正常的，不要反复重试。迁移完成后手动创建 migration 文件并插入 `_prisma_migrations` 记录。
- **editToken 列保留生成逻辑，只删路由和输出**：该列是 `String @unique` 非空，`createUniqueEditToken()` 必须留在 `lib/code.ts`，import 路由继续生成。能删的只有：`/edit/[token]` 页面、export 的 edit 链接列、admin 页面的 editToken 展示。彻底清掉这列就单独做一次改 `String?` 的迁移，不要和本次混。
